Modern Application Development - Microservices

Microservices Security

Authentication and Authorization in a Microservice Architecture

  • Challenge: maintain identity and access management through independent services
  • Monolithic Applications:
    • ​Single server/process holds all user data
    • Session is created when user is authenticated
  • ​Session based (stateful) authentication does not work well with microservices.
  • Alternatives:
    • ​SSO solution
    • Distributed Session
    • Client Side Token

Authentication and Authorization: SSO

Authentication and Authorization: SSO

  • Technology: CAS, SAML
  • Advantages:
    • Good library support
    • User login state is opaque
  • Drawbacks:
    • Stateful: session based
    • Chatty: every public facing service must talk to the authentication server
    • Authentication server is SPOF and potential bottleneck
    • Fairly complex to implement

Authentication and Authorization: Shared User Session

Authentication and Authorization: Shared User Session

  • Technology: Distributed Cache-JDG
  • Advantages:
    • User login state is opaque
    • Highly available and scalable solution
  • Drawbacks:
    • Custom implementation
    • User Session data store needs to be protected

Authentication and Authorization: Client Side Token

Authentication and Authorization: Client Side Token

  • Technology: JWT, OpenID Connect - RHSSO
  • Advantages:
    • Good library support
    • Scalable - Token is self contained
    • Completely stateless
  • Drawbacks:
    • Logout is hard to implement
      • ​Short TTL + Refresh token
    • User login state is not opaque

Authentication and Authorization: Client Side Token with API Gateway

Authentication and Authorization: Client Side Token

  • Technology: JWT, OpenID Connect - RHSSO
  • Advantages:
    • User login state is opaque
    • Logout easy to implement
  • Drawbacks:
    • API Gateway requires custom development 

Json Web Token (JWT)

  • Standard - IETF RFC 7519
  • Compact and self-contained way to transmit information between parties
  • Information is digitally signed
  • Consists of:
    • ​Header: declares the type and the hashing algorithm
    • Payload
      • ​Reserved claims: predefined by the standard (examples: iss, exp)
      • Public/Private claims: can be freely defined
    • Signature
  • Types: Access Token, Identity Token, Refresh Token

Red Hat SSO and OpenId Connect

  • OpenId Connect : authentication protocol which extends OAuth 2.0
  • Makes heavy use of JWT
  • Red Hat Single Sign-on (RHSSO) supports OpenId Connect
    • Authentication flow - involves authorization code which is to be traded against an access token and a refresh token.
    • Implicit flow - access token is directly returned to the user agent (browser)
    • Direct Password Grant - - mostly for testing and legacy applications

Red Hat SSO and OpenId Connect

  • Use case: Authentication & Authorization
    • ​Application asks RHSSO to authenticate a user for them.
      After a successful login, the application will receive an identity token and an access token.
    • ​Identity token contains information about the user (username, email, and other profile information)
    • The access token is digitally signed by the realm and contains access information (user role mappings)
    • The application uses the access token to determine what resources the user is allowed to access
    • The access token can be passed to backend services - Bearer token

Red Hat SSO and OpenId Connect

  • Use case: Delegation - client wants to gain access to remote services on behalf of a user.
    • The client asks RHSSO to obtain an access token.
    • RHSSO authenticates the user, asks the user for consent to grant access to the client.
    • The client then receives the access token. This access token is digitally signed by the realm.
    • The client can make REST invocations on remote services using this access token.

More Security Considerations

  • HTTPS everywhere
  • Mutual TLS - Client-side certificates
    • Payload encryption
    • Payload integrity
    • Server guarantee
    • Caller guarantee
  • Confused Deputy Problem
    • Verify user identity associated with access token
